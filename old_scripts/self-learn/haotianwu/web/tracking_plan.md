# 标注追踪方案说明

这份文档解释：为什么你现在还不能可靠追踪“每个用户改了什么”，以及后面应该怎么升级。

---

## 1. 当前是否已经支持追踪

### 简短答案

**不完全支持。**

当前系统只能说：

- 前端页面上有“用户名”概念；
- 数据库里有“标注记录”；

但不能可靠保证：

- 这条记录到底属于哪个真实用户；
- 某个用户历史上改过哪些字段；
- 某条标注之前被谁改过、改了几次。

---

## 2. 当前系统已经具备哪些基础

数据库 `annotations` 表已经预留了：

- `annotator_id`
- `created_at`
- `updated_at`

这说明数据库设计方向本身是支持用户追踪的。

---

## 3. 为什么现在还不能可靠追踪

### 3.1 用户名只存在 localStorage

当前 `UserContext.tsx` 里的用户名只是浏览器本地字符串。

它不是：

- Supabase Auth 用户
- 数据库可信身份
- 可跨设备统一识别的账号

### 3.2 前端没有写入真实用户 ID

当前保存 annotation 时传的是：

- `annotator_id: null`

因此数据库记录不到真实提交者。

### 3.3 目前没有历史表

当前 `annotations` 更像：

- 当前状态表

而不是：

- 完整历史表

这意味着就算以后接入真实用户，也仍然只能看到“当前结果属于谁”，不一定能完整回放每次改动。

---

## 4. 如果你要“追踪”，至少要做到什么

### 第一层：真实身份追踪

接入 Supabase Auth。

目标：

- 每个用户有真实账号
- 每条 annotation 写入 `annotator_id = auth.uid()`
- 数据库和 RLS 能真正知道是谁在操作

这样你就能回答：

- 这条标注当前是谁提交的

### 第二层：历史追踪

增加一张历史表，例如：

- `annotation_history`

每次 insert / update / delete annotation 时，都记录：

- annotation_id
- gene_id
- field_name
- old_verdict
- new_verdict
- old_corrected_value
- new_corrected_value
- old_comment
- new_comment
- changed_by
- changed_at
- action_type

这样你就能回答：

- 这个字段先后被谁改过
- 每次修改改了什么
- 某个用户历史上总共做过哪些改动

---

## 5. 实时同步和追踪的关系

### 追踪 ≠ 实时同步

你要追踪，核心是：

- 真实身份
- 历史记录

实时同步只是增强体验，不是追踪的前提。

### 如果想让别人改完立刻同步回来

可以再加：

- Supabase Realtime

即对 `annotations` 表做订阅。

这样别人更新后，你当前打开的详情页可以自动刷新相关数据。

---

## 6. 推荐升级顺序

### 阶段 1：先把身份做对

先接入：

- Supabase Auth
- 正确写入 `annotator_id`
- 页面展示当前登录用户

### 阶段 2：再做历史审计

新增：

- `annotation_history`
- 或数据库 trigger / audit log

### 阶段 3：最后做实时体验

新增：

- Realtime 订阅
- 自动刷新当前页面标注

---

## 7. 你最终应该能得到什么能力

做完后，系统应该能回答：

1. 这条标注是谁提交的？
2. 这条标注最后一次什么时候修改？
3. 这个字段历史上被哪些人改过？
4. 某个用户总共改了哪些论文、哪些基因、哪些字段？
5. 别人的新标注是否能实时同步到当前页面？

如果这五个问题都能回答，才算真正进入“多人可追踪协作标注”阶段。
